# Layout이 온디스크 이름·경로 규칙을 단독 소유한다 — 그러나 HTTP 라우트는 소유하지 않는다

`.objects` · `.tmp-` · `.meta.json` · `.bucket.json` · `.gc-pending.json` · `.corrupt`
같은 온디스크 이름 규칙이 store 내부 4개 파일과 reconcile·http/state에 흩어져 있어,
같은 문법·우선순위·재귀 전략이 여섯 곳에서 각자 재유도되고 있었다. 접두사 하나만
어긋나도 중단된 업로드의 temp 파일이 `Other`로 분류돼 정리를 회피하는 식의 조용한
데이터 사고가 가능한 구조였다. 그래서 `src/layout.rs`를 **온디스크 이름·경로 규칙의
단일 소유자**로 세우고(이름을 만드는 쪽: 경로 저작·검증, 읽는 쪽: 분류기·커밋 포인터
워커), 모든 소비자를 그 seam으로 통과시켰다. **Layout을 거치지 않은 경로 저작은 규칙
위반이다.**

**그러나 예약 이름의 "이름"만 Layout의 것이고, 그 이름이 공개 표면에서 갖는 "라우트
모양"은 Layout의 것이 아니다.** axum 라우트 패턴(`/api/{*rest}`)은 온디스크 지식이
아니기 때문이다. 이 경계는 다음 항목에서 되풀이되기 쉬우므로 명시해 둔다.

## Considered Options

- **예약 라우트를 `layout::RESERVED_BUCKETS`에서 루프로 파생** — 원래 계획이었고,
  기각됐다. wire 레벨 실측 결과 **행위를 깬다**: 현재 세 예약 라우트는 모양이
  비대칭이고(`api`는 서브트리 `/api/{*rest}`, `healthz`·`readyz`는 정확 일치),
  **그 비대칭이 곧 관측 행동**이다. 균일 파생은 `/healthz/{*rest}`·`/readyz/{*rest}`를
  새로 만들어, 지금 catch-all(`GET` 전용)에 걸려 **405 + `Allow: GET,HEAD`**를 내는
  `PUT /healthz/foo` 류를 404로 바꿔버린다(상태코드 8셀 + 바디 2셀). 이름만 담은
  목록에서 모양의 비대칭은 **파생 불가능**하다.
- **Layout이 이름과 라우트 패턴을 함께 소유** — 기각. 한 쌍의 누수를 다른 쌍으로
  갈아끼울 뿐이다. Layout의 헌장은 **온디스크** 이름이지 HTTP 라우팅이 아니다.
- **채택: 관심사 분리 + 기계 결속.** 이름은 `layout::RESERVED_BUCKETS`, 모양은
  `public.rs`의 `RESERVED_ROUTES`, 그리고 **두 목록의 정합성을 테스트가 강제**한다
  (정방향: 모든 예약 버킷명에 그림자 라우트가 존재 / 역방향: 모든 그림자 라우트가
  예약 버킷명에 대응). 드리프트는 조용한 버그가 아니라 테스트 실패가 된다.

## Consequences

- **`public.rs`의 `RESERVED_ROUTES`가 `RESERVED_BUCKETS`와 겹쳐 보이는 것은 의도된
  것이다.** "중복이니 파생으로 정리하자"는 유혹이 정확히 위에서 기각된 그 실수다.
  라우트의 **모양**(정확 일치냐 서브트리냐)은 하위 경로 비-GET 응답이 404냐 405냐를
  결정하므로 기계가 파생할 수 없고 사람이 골라야 한다. `reserved_route_shape_asymmetry_is_load_bearing`
  테스트가 이 행동을 핀하고 있어, 바꾸려면 이제 의도적·가시적으로 해야 한다.
- 온디스크 이름 리터럴은 **테스트에서는 raw 문자열로 유지**한다. layout 상수를
  경유시키면 동어반복이 되어 회귀 감지력을 잃는다(테스트가 온디스크 바이트를 핀하는
  것이 존재 이유다).
- `Layout::root()`는 경로 **저작**이 아니라 베이스 디렉터리 노출이다(루트 열거용).
  root는 온디스크 이름 규칙이 아니라 config 값이므로 이 노출이 seam을 해치지 않는다.
